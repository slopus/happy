## WARNING Fresh local Codex exits are masked by thread discovery
**File:** packages/happy-cli/src/codex/codexLocalLauncher.ts:173
**Issue:** For a fresh local launch with no known `codexThreadId`, `launchNativeCodex` always waits for `discoveryPromise` after the child exits, and throws the discovery error if no session metadata appears. If native `codex` exits before writing a session file, for example because startup flags are rejected, config is invalid, auth fails, or the user exits before a session is created, Happy waits up to the discovery timeout and then reports `Could not discover Codex thread id` instead of returning the child exit code.
**Recommendation:** When `exitPromise` wins before discovery and no handoff is pending, return `{ type: 'exit', code }` without requiring discovery. Keep discovery mandatory only for handoff/switch paths where remote continuation needs a thread id, and optionally preserve a discovered id if it was already found before exit.
**Rationale:** Fresh local mode should behave like the native TUI when it simply exits or fails to start. Masking the actual native exit path with a later discovery failure makes startup failures misleading and can cause the terminal command to reject rather than exiting with the Codex process status.
