## [WARNING] Native local Codex bypasses configured Happy sandbox
**File:** packages/happy-cli/src/codex/codexLocalLauncher.ts:110
**Issue:** `launchNativeCodex` spawns `codex` directly with inherited stdio/env and no Happy sandbox wrapping. `runCodex` still reads `sandboxConfig` and the remote app-server path passes it into `CodexAppServerClient`, but the new local path does not pass or apply that config before launching native Codex. Since terminal-started Codex now defaults to local mode, users with Happy sandbox enabled can get an unsandboxed native Codex process unless they explicitly force remote mode.
**Recommendation:** Thread `sandboxConfig` into the local launcher and wrap the native Codex command with the existing sandbox manager, matching the Claude local path, or force remote mode when Happy sandboxing is enabled. If sandbox initialization fails, fail closed or require an explicit `--no-sandbox` override.
**Rationale:** This is a sandbox/permission mapping regression: a user-configured confinement boundary can be silently bypassed by the new default local launch path, giving Codex broader filesystem and network access than the Happy session metadata advertises.
