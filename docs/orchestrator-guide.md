# Orchestrator 提示词示例集

> 在 Happy 会话中直接发送以下提示词，AI 会自动调用 Orchestrator 派发任务。

---

## 一、并行执行类

### 1. 全包类型检查

```
帮我同时检查所有包的类型错误：happy-server 用 yarn build，happy-app、happy-cli、happy-wire 用 yarn typecheck。并行跑，汇总结果给我。
```

### 2. 多视角 Code Review

```
派发三个任务同时审查 happy-server 的 orchestratorRoutes.ts：
- 一个看安全性（注入、权限、输入验证）
- 一个看性能（N+1 查询、不必要的 await）
- 一个看逻辑正确性（边界条件、竞态、错误处理）
分别安排给 Claude、Codex、Gemini，最后汇总。
```

### 3. 批量生成测试

```
安排并行任务，为 happy-server 的以下路由文件生成单元测试：
- orchestratorRoutes.ts
- sessionRoutes.ts
- authRoutes.ts
每个文件派发给一个 Codex，参考现有测试风格。
```

## 二、DAG 依赖链类

### 4. 设计→实现→验证 流水线

```
帮我开发一个"用户数据导出"功能，按这个顺序安排：
1. 先派发给 Claude 设计 API 接口（Zod schema，支持 CSV/JSON）
2. 设计完成后，同时派发给 Codex 实现后端（happy-server）和前端（happy-app）
3. 前后端都完成后，跑一遍类型检查验证

用 orchestrator 编排这个流水线。
```

### 5. 调研→决策→实施

```
我想给项目加一个实时通知功能，但不确定用什么方案。帮我安排：
1. 同时派发两个调研任务：一个调研 Socket.IO，一个调研 SSE
2. 调研都完成后，再派发一个任务根据结论选方案并实现

用 DAG 依赖来保证顺序。
```

### 6. 重构三步走

```
我想把 happy-server 的 orchestratorRoutes.ts 拆成更小的模块。请安排：
1. 先分析当前文件结构，输出拆分方案
2. 方案确定后执行拆分
3. 拆分完成后跑 yarn build 验证没有类型错误
```

## 三、跨 Provider 协作类

### 7. 让不同 AI 比拼方案

```
分发同一个任务给 Claude、Codex、Gemini：各自设计一个 REST API 的分页方案，独立完成，最后我来对比选择。
```

### 8. 各取所长的分工

```
安排以下任务：
- 派发给 Claude opus：为 happy-voice 设计新的 Provider 抽象层
- 派发给 Codex gpt-5.4-xhigh：根据设计结果编写代码
- 设计先完成，实现再开始
```

## 五、异步 + 追踪类

### 11. 后台执行大任务

```
用 async 模式派发一个大重构任务给 Codex：把 happy-cli 里所有的 console.log 替换成统一的 logger 工具。提交后告诉我 runId，我稍后来查结果。
```

### 12. 查看和管理运行

```
帮我看看现在有哪些正在运行的派发任务。
```

```
等待 runId 为 xxx 的任务完成，完成后把结果总结给我。
```

```
取消 runId 为 xxx 的运行，原因是方向变了。
```

### 13. 对结果追加要求

```
上次那个任务做得不错，但少了错误处理。帮我对 taskId xxx 追加消息：请补充完整的错误处理和边界情况测试。
```

## 六、实际开发场景类

### 14. PR 前全套检查

```
我准备提 PR 了，帮我安排并行检查：
1. ESLint 检查
2. TypeScript 类型检查（所有包）
3. 单元测试
4. lint 和类型检查都通过后，再跑集成测试
汇总所有结果。
```

### 15. 新功能完整开发

```
帮我安排开发一个"会话收藏"功能：
1. 先派发给 Claude 做接口设计（API + 数据库 schema）
2. 设计完后并行派发：Codex 写后端、Codex 写前端
3. 前后端都完成后：派发 Claude 做 Code Review，同时派发 Codex 跑测试
全程用 DAG 自动编排。
```

### 16. 数据库迁移流水线

```
帮我编排数据库迁移流程：
1. 先备份数据库
2. 备份成功后执行 prisma migrate deploy
3. 迁移成功后验证新表结构
如果迁移失败，自动重试最多 3 次。
```

### 17. 文档批量生成

```
安排并行任务为 happy-server 的所有路由文件生成 API 文档：
- authRoutes.ts
- sessionRoutes.ts
- orchestratorRoutes.ts
- projectRoutes.ts
每个文件派发一个任务，maxConcurrency 设为 3，每个任务带 retry。
```

### 18. Bug 排查

```
有个 bug：用户登录后偶尔 token 失效。帮我同时安排三个方向排查：
- 检查 token 生成和验证逻辑
- 检查 session 过期和刷新机制
- 检查相关的中间件和错误处理
并行调查，汇总发现。
```

## 七、进阶用法

### 19. 指定模型和思考级别

```
派发给 Codex gpt-5.4-xhigh（最高思考级别）分析 happy-server 的 scheduler.ts 有没有竞态条件。
```

```
安排两个并行任务：派发给 Claude opus 审查认证流程的安全性，派发给 Gemini 3 Pro 调研业界最佳实践。
```

### 20. 带重试的不稳定任务

```
安排一个任务调用外部 API 抓取数据并保存：
- 超时设为 30 秒
- 最多重试 3 次，每次间隔 5 秒
- 抓取成功后再派发一个任务处理数据
```
